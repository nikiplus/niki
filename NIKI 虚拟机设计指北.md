# NIKI 虚拟机设计指北：从源代码到字节码的物理映射

要设计虚拟机，首先必须明确其物理边界。
虚拟机的本质是一个翻译与执行的黑盒：

> **[输入] 源代码 -> 虚拟机 -> [输出] 可执行的字节码 / 机器码**

所谓的“源代码”，是我们用强逻辑定义的文本。为了直观理解，我们可以把代码块视为**便利贴上写的字**，而代码文件就是一个贴满便利贴的**小黑板**。

假设有这样一个文件：

```niki
// 便利贴1 [struct]: 属性值
struct 属性值 {
    hp = 100;
    mp = 20;
    atk = 10;
}

// 便利贴2 [class]: 玩家基类
class 玩家基类 {
    属性值; // 连接到便利贴1

    if (属性值.hp <= 0) {
        print("玩家死亡");
    }
    return 属性值.hp;
}
```

这段文本对人类友好，但对计算机而言毫无意义。CPU 的底层只认 0 和 1。要让计算机执行这段逻辑，我们必须进行“翻译”。

---

## 计算机语言的物理降维：从纸带到指令集

在讨论翻译机制前，我们需要理解目标语言（机器语言）的演进逻辑。

最早的计算机是纸带打孔机：打孔为 `0`，不打孔为 `1`。通过组合不同的 0 和 1，我们可以映射不同的硬件操作。这就是计算机语言的第一层抽象——**“指令（Instruction）”**。

无论 CPU 架构多么复杂，其核心指令集通常只包含以下几类：

### 1. 算数与逻辑运算 (ALU)
CPU 的“计算器”。
- **算数**：`ADD`, `SUB`, `MUL`, `DIV`
- **逻辑与位移**：`AND`, `OR`, `SHL`, `SHR`

### 2. 数据传输 (Data Movement)
在寄存器和内存间搬运数据。
- **内存交互**：`LOAD` (读), `STORE` (写)
- **寄存器与栈**：`MOV` (复制), `PUSH` / `POP`

### 3. 程序流控制 (Control Flow)
打破代码从头到尾的顺序执行，实现 `if` 和 `loop`。
- **跳转**：`JMP` (无条件), `JZ` / `JNE` (条件跳转)
- **函数栈**：`CALL` / `RET`

### 4. 比较与位测试 (Compare & Test)
配合条件跳转使用。
- **`CMP`**: 减法比较，影响标志位。
- **`TEST`**: AND 运算测试。

### 5. 高级/特殊指令 (System & SIMD)
- **`SIMD`**: 单指令多数据（如同时处理 4 个浮点数），物理引擎和 DOD 架构的性能核心。
- **`SYSCALL`**: 陷入内核，请求操作系统介入。

这套指令的集合，构成了第二层抽象——**“指令集（Instruction Set）”**。基于指令集的助记符，诞生了“汇编语言”。
在此之上，为了让人类摆脱物理寄存器的束缚，诞生了第三层抽象——**“高级语言（C++, Java, NIKI）”**。

## 逆向解析：拆解虚拟机黑盒

我们要将源代码翻译成机器码，本质上就是在**逆走上述的历史进程**。我们需要设计一个翻译器，将高级语言降维成底层的指令流。

但虚拟机不能直接“阅读”整块文本。它必须先观测、切分，然后才能理解。
因此，我们将黑盒拆解出第一层物理结构：

> **[输入] 源代码 -> 扫描器 (Scanner) -> Token 流 -> 翻译层 -> [输出] 字节码**

### 扫描器 (Scanner) 与 Token
扫描器的唯一任务，是将无结构的字符串文本，切分成一个个离散的、有意义的词元——**Token**。
Token 可以是关键字（`struct`）、标识符（`hp`）、运算符（`<=`）或常量（`100`）。

**为什么要切分 Token？**
计算机科学是一门**用有限的规则，描述无限信息的艺术**。
自然语言是无限且随意的，而机器只能处理有限且结构化的输入。Token 就是我们为了收敛这无序的文本，而定义的“最小语法原子”。

**工业级编译器的按需扫描（On-Demand Scanning）**
在真实的物理内存中，我们绝对**不能**将一个 10MB 的源代码文件一次性全部扫描并存入一个巨大的 `std::vector<Token>` 中。
- **内存灾难**：这会瞬间吃掉几十兆的连续内存，引发严重的堆碎片化。
- **性能浪费**：如果代码第一行就有语法错误，扫描后续的几万行毫无意义。
- **DOD 架构实践**：Scanner 被设计为一个**迭代器（Iterator）**。它不保存整个 Token 数组，而是像流水线上的阀门一样，下游（Parser）每调用一次 `scanToken()`，它就向前滑动指针（`current++`），返回当前的句柄。同一时间存在于内存中的，永远只有当前正在解析的极少数 Token。

但注意：扫描器仅仅是将文本打散成了 Token 流，它并不理解 `hp = 100` 是一个赋值操作，它只是看到了 `hp`, `=`, `100` 三个孤立的词。它解决了“字”的问题，但没有解决“语法”的问题。

## 建立维度：解析器（Parser）与抽象语法树（AST）

要理解这些离散词元的组合逻辑，我们必须进入虚拟机的下一层——**解析器（Parser）**。

> **[输入] 源代码 -> 扫描器 (Scanner) -> Token 流 -> 解析器 (Parser) -> AST语法树 -> 编译器 (Compiler) -> [输出] 字节码 -> 虚拟机执行 (VM Execution)**

### 解析器 (Parser) 的核心使命
如果说 Scanner 是在做“一维的切片”，那么 Parser 就是在做**“二维/树状的组装”**。
Parser 的任务是不断向 Scanner 索要 Token，并根据语言的语法规则（如加减乘除的优先级），将这些 Token 组装成一棵树——**抽象语法树（AST, Abstract Syntax Tree）**。

以 `1 + 2 * 3` 为例：
Scanner 看到的只是一维序列：`[1] [+] [2] [*] [3]`。
如果直接从左到右执行，结果是 `(1+2)*3 = 9`，这在数学上是错误的。

Parser 必须理解“优先级（Precedence）”，并将这串序列提升为树形结构：
```text
      +
     / \
    1   *
       / \
      2   3
```
在这棵树中，越底层的节点（`*`）优先级越高，越先被执行。这就是 Parser 存在的根本物理意义：**它将人类习惯的中缀表达式，转化为了计算机可以直接自底向上遍历的执行树。**

### 信息擦除与意图升维（Information Erasure & Intent Elevation）
从 Token 流到 AST，信息并不是 1:1 无损传递的。观察上面的树状图，你会发现源代码中的**括号 `()` 消失了**——它们在物理结构上被擦除了，转而隐式地化为了树的层级深度（深度即优先级）。

这就引出了一个令初学者极易混淆的问题：**为什么 Token 和 OpCode (指令) 中，往往存在看似一模一样的枚举（例如 `TOKEN_PLUS` 和 `OP_ADD`）？**
注意为什么我提出这个问题，因为二者只是看起来像，但却实际上有许多不同，而这份不同是极为重要的。

既然它们看起来一样，为什么我们要写两套枚举，而不是一套用到头？
因为它们所处的**物理维度**和**工程职责**存在不可逾越的鸿沟：
- **Token 是“表象（Syntax）”**：它仅仅是对源代码字符串的**客观记录**。`TOKEN_PLUS` 只是说“这里有一个十字形状的墨水痕迹”。
- **OpCode 是“意图（Semantics/Execution）”**：它是虚拟机必须执行的**物理动作**。`OP_ADD` 是在命令 CPU“把寄存器 A 和 B 的值弹出来，送入 ALU 进行加法运算，再把结果压回去”。

**Parser 和 Compiler 的核心在于此：**
它们联手完成了一次从“文字表象”到“操作意图”的**惊心一跳**。
在 `a + b` 这个例子中，Parser 看到了 `TOKEN_PLUS`，结合上下文，它确信这是一个加法运算，并在 AST 中生成一个加法节点；随后，Compiler 遍历这个节点，最终发射出 `OP_ADD` 指令。

这个过程中主动丢失了大量对执行毫无帮助的“视觉信息”（如空格、换行、括号、注释），就如同提取人脸特征时滤除了光影和背景。只有褪去语法（Syntax）的伪装，程序的执行语义（Semantics）才能在物理内存中裸奔。这是整个编译器工程中最具哲学美感的一次思维跃迁。


### 类型系统的物理意义：防止比特流的灾难性误读
从“表象”到“意图”的转换，伴随着一个致命的风险：**信息的丢失**。

始终牢记，在 CPU 的物理寄存器和内存中，不存在 `1` (整数)、`1.0` (浮点数) 或 `"1"` (字符串) 的区别。它们最终都会被降维成一串毫无特征的 `0` 和 `1` 的比特流（Bitstream）。
如果你在 AST 转换到字节码的过程中，没有明确记录这些数据的**类型（Type）**，虚拟机就会在运行时变成一个盲人。它可能把一段代表字符串的内存地址，当成一个浮点数送入 ALU 进行乘法计算，这将直接导致程序崩溃或产生灾难性的“脏数据”。

因此，谁来负责记录和检查这些类型，就衍生出了现代编程语言的两大流派：

1. **静态类型（Static Typing，如 C++, Rust）**
   - **机制**：在**编译期（Compile Time）**，也就是在 Parser 或 Compiler 阶段，由编译器对整个 AST 进行严格的类型推导和检查（Type Checking）。
   - **物理代价**：编译器代码极度复杂，编译时间变长。
   - **运行时优势**：一旦编译通过，生成的字节码（机器码）是“纯粹”的。虚拟机在执行 `OP_ADD_INT` 时，**完全不需要**检查左右两边是不是整数，直接无脑计算。这是极致性能的保障。

2. **动态类型（Dynamic Typing，如 JavaScript, Python）**
   - **机制**：编译器在生成字节码时“偷懒”，遇到加法统统生成一个模糊的 `OP_ADD`。类型检查被推迟到了**运行期（Runtime）**。
   - **物理代价**：在虚拟机执行 `OP_ADD` 时，必须先在内存中读取左右两个变量的“类型标签（Type Tag）”，判断它们是数字还是字符串，如果是字符串还要调用拼接函数，如果是数字还要判断是整数还是浮点数。
   - **性能灾难**：原本 CPU 只需要 1 个时钟周期就能完成的物理加法，在动态类型语言中可能需要几百个时钟周期来进行条件分支跳转（Branching）和类型分发（Dispatch）。

对于 NIKI 这样一个追求极致性能、采用 DOD 架构的引擎语言来说，我们注定要走向**强静态类型**的道路。这意味着我们的 Parser 和 Compiler 必须承担起最沉重的类型推导工作，以换取 VM 在运行时的轻装上阵。


### AST 的物理陷阱与 DOD 破局
在传统的 OOP（面向对象）教科书中，AST 的每个节点往往是一个在堆上通过 `new` 分配的对象（如 `new BinaryExpr(left, right)`）。
**这是严重的性能毒药。**
当树的层级达到几千层时，内存中会散布着无数通过指针相连的小碎片。CPU 在遍历这棵树时，会触发海量的缓存未命中（Cache Miss）。

**NIKI 的架构选择**：
我们将坚决摒弃“指针跳转”的树结构。我们要用数组的索引（Index/Handle）来代替物理指针，将 AST 拍平在一块连续的内存池（Arena Allocator）中。这才是符合 DOD（面向数据设计）的工业级做法。

## 最终闭环：编译器（Compiler）与虚拟机（VM）

当我们有了一棵结构正确的 AST 后，接下来的任务就是**遍历它**，并生成最终的 0 和 1（字节码）。



### 虚拟机（VM）的形态
NIKI 最终要执行的“字节码”，本质上就是一个巨大的 `std::vector<uint8_t>` 数组。
虚拟机（VM）的核心，就是一个包含巨大 `switch-case` 语句的死循环：
1. 从数组中读取当前指令（`fetch`）。
2. 根据指令类型执行操作（`decode & execute`）。
3. 操作虚拟机的栈（Stack）或寄存器（Registers）。
4. 移动指令指针（`Instruction Pointer, IP`），循环往复。

### 额外思考
你看，我们的演进过程中，事实上就是在一层又一层的给高级语言到底层机器码之间添加各类抽象层级。
